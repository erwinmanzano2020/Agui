# AGENTS.md

## Scope
This file applies to HR planning, milestone, freeze, and related documentation under `/docs/hr`.

Use this file together with the root `/AGENTS.md`. Root policy is canonical.

---

## Why This File Exists
HR is the current active delivery stream.

These documents are not casual notes. They often act as planning contracts, freeze references, and scope boundaries. This file exists to prevent documentation edits from accidentally expanding scope or softening non-negotiables.

---

## Status Labeling Discipline
When editing documents in this subtree, keep status explicit.

Use and preserve clear status labels such as:
- active
- frozen
- proposed
- deferred
- historical

Do not blur status by mixing approved scope with future ideas.

---

## Freeze-Safe Editing Rules
- Preserve frozen boundaries unless the task explicitly authorizes a contract change.
- Do not quietly widen milestone scope.
- Do not rewrite history to make a future idea look approved.
- When a document includes non-negotiables, preserve them clearly.
- When changing active milestone docs, cross-check the master plan and current freeze context.

---

## Contract vs Implementation Note Separation
Keep these distinct:
- contract statements
- implementation notes
- future ideas
- risks
- follow-up work

Do not present a tentative idea as if it is already part of the approved milestone contract.

---

## Cross-Reference Discipline
When editing milestone or scope docs, link or reference the relevant source of truth where needed:
- HR master plan
- roadmap
- operating principles
- freeze or milestone tracking docs

The goal is to preserve clarity about what is approved, what is active, and what is future work.

---

## Language Rules
Use language that is:
- explicit
- phase-aware
- contract-safe
- non-ambiguous

Avoid wording that implies approval when none exists.

Examples of risky wording to avoid unless explicitly true:
- “will also include”
- “for now we can just add”
- “later-ready” when it changes current contract assumptions
- “small extension” when it expands scope

---

## Escalate Instead of Editing Through It
Stop and surface the issue if the requested doc change appears to:
- alter active milestone contract meaning
- weaken HR-first discipline
- introduce future-phase scope into active HR work
- reinterpret identity or tenancy rules without approval
- conflict with roadmap or master plan
